## Design Context

### Users
Systems engineers, OCSMP certification candidates, MBSE practitioners, and SysML trainers running cohorts. Two modes of use: solo self-study (needs clear progression, instant feedback) and team/training contexts (instructors leading exercises, students working in parallel). Users are technically fluent — they are comfortable with IDEs, UML tooling, and engineering notation. They do not need hand-holding, but they do need clarity and confidence in an unfamiliar UI.

### Brand Personality
**Technical. Sharp. Inviting for learners.**

The voice is that of a rigorous senior engineer who is also a good teacher — precise without being cold, demanding without being intimidating. Copy should be direct and specific (cite standards, diagram types, real engineering domains). Never dumbed down, never gamified, never corporate-boring.

### Aesthetic Direction
- **Landing**: Bold editorial — pill collage headlines, dot-grid paper background (`#F5F4F0`), electric blue (`#1B1BFF`) as the single dominant accent. High contrast. Confident typography (`font-black`).
- **Workspace**: IDE precision — light mode only, zinc-based panels, `border-2 border-zinc-200` dividers, `font-mono` for labels and technical notation. The canvas area uses a dark theme (`#1e1e1e`) as a deliberate contrast zone — it is where engineering work happens.
- **Red dots** in the hero are decorative noise only — red is NOT a brand color. Keep red reserved for error/warning states exclusively.
- **No dark mode** for the exercise workspace. Light mode only throughout.

**Anti-references**: Not a gamified learning app (no XP bars, achievement popups, or cartoon mascots). Not a passive lecture tool (no slide decks or read-only content). Not enterprise-bland (no navy/grey dashboards, no modal-heavy admin UI).

**References in spirit**: The precision of Linear or Vercel's dashboard, the editorial boldness of Are.na or Framer's marketing, the IDE familiarity of VS Code.

### Design Principles

1. **Precision over decoration.** Every visual element must earn its place. Spacing, borders, and typography choices should communicate structure — not decorate. If it doesn't help a user understand the diagram or the task, remove it.

2. **Dual register, consistent identity.** The landing is editorial and bold; the workspace is technical and restrained. Both use the same type scale, accent color, and border language. Users should feel continuity, not jarring mode-switching.

3. **Learner confidence through clarity.** Feedback (grading results, validation console, hints) must be legible at a glance. Error states use red. Success states use green. The blue accent (`#1B1BFF`) is reserved for task prompts, CTAs, and primary interactive elements — it signals "this matters."

4. **Technical credibility in every detail.** Use `font-mono` for diagram notation, node labels, stage indices, and scores. Use correct SysML terminology everywhere (Block, Port, Allocation — not "item" or "thing"). The UI should feel like it was built by someone who actually knows SysML.

5. **Light, capable, no clutter.** No unnecessary modals, no drawer-within-drawer nesting, no explanatory text that repeats what the label already says. The interface should get out of the way once the user understands it — which should happen within the first exercise.
